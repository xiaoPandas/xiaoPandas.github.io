μC/OS-II的中断和时钟

主要内容：

  • μC/OS-II 的中断管理和中断服务程序的结构；

  • μC/OS-II 的系统时钟及实现方法；

  • μC/OS-II 的时间管理服务：延时，取消延时；

临界段

在应用程序中经常有一些代码段必须不受任何干扰地连续运行，这样的代码段叫做临界段。

为了使临界段在运行时不受中断所打断，在临界段代码前必须用关中断指令使 CPU 屏蔽中断请求，而在临界段代码后必须用开中断指令解除屏蔽使得 CPU 可以响应中断请求。

由于各厂商生产的 CPU 和 C 编译器的关中断和开中断的方法和指令不尽相同，为增强 μC/OS-II 的可移植性（即在 μC/OS-II 的各个 C 函数中尽可能地不出现汇编语言代码)，μC/OS-II 用两个宏来实现中断的开放和关闭，而把与系统的硬件相关的关中断和开中断的指令分别封装在这两个宏中：
OS_ENTER_CRITICAL()    OS_EXIT_CRITICAL()

临界代码保护方式，即上述两个宏的实现方法有 3 种：

  • OS\_CRITICAL\_METHOD = 1
    直接使用处理器的开中断和关中断指令来实现宏，简单高效，但无法满足嵌套的情况；

  • OS\_CRITICAL\_METHOD = 2
    可以使 CPU 中断允许标志的状态，在临界段前和临界段后不发生改变；把 CPU 的允许中断标志保持到堆栈中，然后再关闭中断，这样在临界段结束时，只要把堆栈中保存的 CPU 允许中断状态恢复就可以了；

  • OS\_CRITICAL\_METHOD = 3
    在关中断前，使用局部变量保存中断状态；

系统响应中断的过程

  • 系统接收到中断请求；

  • 这时如果 CPU 处于中断允许状态(即中断是开放的)，系统就会中止正在运行的当前任务，而按照中断向量的指向转而去运行中断服务子程序；

  • 当中断服务子程序的运行结束后，系统将会根据情况返回到被中止的任务继续运行或者转向运行另一个具有更高优先级别的就绪任务(进行一次任务调度)。

中断级任务切换函数

OSIntCtxSw()：在中断服务程序中调用的负责任务切换工作的函数；

μC/OS-II的系统时钟

μC/OS-II 与大多数计算机系统一样，用硬件定时器产生一个周期为 ms 级的周期性中断来实现系统时钟，最小的时钟单位就是两次中断之间相间隔的时间，这个最小时钟单位叫做时钟节拍(Time Tick)。

OSTickISR() & OSTimeTick()

硬件定时器以时钟节拍为周期定时地产生中断，该中断的中断服务程序叫做 OSTickISR()。

中断服务程序通过调用函数 OSTimeTick() 来完成系统在每个时钟节拍时需要做的工作。

任务的延时

由于嵌入式系统的任务是一个无限循环，并且 μC/OS-II 还是一个抢占式内核，所以为了使高优先级别的任务不至于独占 CPU，可以给其他任务优先级别较低的任务获得 CPU 使用权的机会：

μC/OS-II 规定：除了空闲任务之外的所有任务必须在任务中合适的位置调用系统提供的函数 OSTimeDly()，使当前任务的运行延时(暂停)一段时间并进行一次任务调度，以让出 CPU 的使用权。

任务的同步和通信

需要的原因：

  • 多个任务运行时，经常需要互相无冲突的访问同一个共享资源；

  • 或者需要互相支持和依赖；

  • 或者互相加以必要的限制和制约，才能保证任务的顺利运行；

所以多个任务共享同一资源或有工作顺序要求时，正式工作前要先进行通信。

事件

任务间的同步依赖于任务间的通信。在 μC/OS-II 中，是使用信号量、消息邮箱和消息队列这些被称作事件的中间环节来实现任务之间的通信的。

事件控制块

事件控制块 ECB(Event Control Block)：描述诸如信号量、消息邮箱和消息队列这些事件；其中包含了包括等待任务表在内的所有有关事件的数据 。

  • OS_EventTaskWait()：把一个任务置于等待状态。
    该函数通常在任务调用函数 OS×××Pend() 请求一个事件时，被 OS×××Pend() 所调用。
    void OS_EventTaskWait (OS_EVENT *pevent)

  • OS_EventTaskRdy()： 把等待态任务变为就绪态。
    该函数通常在任务调用函数 OS×××Post() 发送一个事件时，被 OS×××Post() 所调用。

  • OS_EventTO()：正在等待事件的任务已经超过了等待时间，未具备可以运行的条件，但要使它进入就绪态。
    该函数通常在任务调用函数 OS×××Pend() 请求一个事件时，被 OS×××Pend() 所调用。
    void OS_EventTO (OS_EVENT *pevent)

空事件控制块链表(OSEventFreeList)：

在 μC/OS-II 初始化时，系统会在初始化函数 OSInit() 中按应用程序使用事件的总数创建 OS_MAX_EVENTS 个空事件控制块并使用成员 OSEventPtr 作为链接指针，把这些空事件控制块链接成一个单向链表。

信号量及其操作

  1. OSSemCreate()：创建信号量，并对计数初值赋值，0-65535；
    OS_EVENT *OSSemCreate (INT16U cnt);
    执行步骤：
    从空闲事件控制块链表中得到一个 ECB；
    初始化 ECB，包括设置信号量的初始值、把等待任务列表清零、设置 ECB 的事件类型等；
    返回一个指向该事件控制块的指针。

  2. OSSemPend()：请求信号量，操作系统的 P 操作，将信号量的值减 1；
    void OSSemPend (OS_EVENT *pevent, INT16U timeout, INT8U *err);
    timeout = 0 表示任务等待时间为无限长；
    执行步骤：
    如果信号量的计数值大于 0，将它减 1 并返回；
    如果信号量的值等于 0，则调用本函数的任务将被阻塞起来，等待另一个任务把它唤醒，
    然后调用 OSSched() 函数，调度下一个最高优先级的任务运行。

  3. OSSemPost()：释放信号量，操作系统的 V 操作，将信号量的值加 1；
    INT8U OSSemPost (OS_EVENT *pevent);
    执行步骤
    检查是否有任务在等待该信号量，如果没有，将信号量的计数值加1并返回；
    如果有，将优先级最高的任务从等待任务列表中删除，并使它进入就绪状态，
    调用OSSched()，判断是否需要进行任务切换。

  4. OSSemDel()：删除信号量；
    pevent：信号量的指针，opt：删除条件选项，err：错误信息。
    OS_EVENT *OSSemDel (OS_EVENT *pevent, INT8U opt, INT8U *err);

任务优先级反转

  • 任务优先级反转：在可剥夺型内核中，当任务以独占方式使用共享资源时，出现的低优先级任务先于高优先级任务而被运行的现象。

  • 互斥型信号量：有一个共享资源，用于保护该资源的信号量在同一时间只允许一个任务以独占的方式对该资源进行访问。

优先级反转出现原因：

使用信号量的任务是否能够运行是受任务的优先级别和是否占用信号量两个条件约束的，而信号量的约束高于优先级别的约束。

于是当出现低优先级别的任务与高优先级别的任务使用同一个信号量，而系统中还存有别的中等优先级别的任务时：

如果低优先级别的任务先获得了信号量，就会使高级别的任务处于等待状态，而那些不使用该信号量的中等级别的任务却可以剥夺低优先级别的任务的 CPU 使用权而先于高优先级别的任务而运行了。

解决方法：

使获得信号量任务的优先级别在使用共享资源期间暂时提升到所有任务最高优先级的高一个级别上，以使该任务不被其他的任务所打断，从而能尽快地使用完共享资源并释放信号量，然后在释放了信号量之后再恢复该任务原来的优先级别。

互斥型信号量

互斥型信号量的事件控制块：

成员 OSEventType 要赋以常数 OS_EVENT_TYPE_MUTEX 以表明这是一个互斥型信号量；

没有使用成员 OSEventPtr；

成员 OSEventCnt 被分成了低 8 位和高 8 位两部分：低 8 位用来存放信号值(该值为 0xFF 时，信号为有效，否则信号为无效)，高 8 位用来存放为了避免出现优先级反转现象而要提升的优先级别 prio。

  • OSMutexCreate()：创建互斥型信号量；

  • OSMutexPend()：请求管理一个独占式共享资源的互斥型信号量；
    如果信号量有信号( OSEventCnt 的低 8 位为 0xFF )，意味着当前无任务占用资源，任务继续运行并对该资源进行访问；否则进入等待状态，直至占用这个资源的其他任务释放了该信号量。

  • OSMutexPost()：释放一个互斥型信号量；

消息邮箱

消息邮箱的事件控制块：

成员 OSEventType 为常数 OS_EVENT_TYPE_MBOX；

把数据缓冲区的指针 &message 赋给成员 OSEventPrt；

成员 OSEventCnt 为 0x0000；

消息邮箱是在两个需要通信的任务之间通过传递数据缓冲区指针的方法来通信的。

一个邮箱可能处于两种状态：

满的状态：邮箱包含一个非空指针型变量；

空的状态：邮箱的内容为空指针 NULL；

邮箱的系统服务：

  • OSMboxCreate()：创建邮箱，需先定义 msg 的初始值。
    一般情况下，初始值为 NULL；也可以事先定义一个邮箱，然后把这个邮箱的指针作为参数传递到函数 OSMboxCreate () 中，使之一开始就指向一个邮箱。
    OS_EVENT *OSMboxCreate (void * msg);

  • OSMboxPost()：向消息邮箱发送消息；

  • OSMboxPend()：任务请求邮箱；
    查看邮箱指针 OSEventPtr 是否为 NULL，如果不是 NULL 就把邮箱中的消息指针返回给调用函数的任务，同时用 OS\_NO\_ERR 通过函数的参数 err 通知任务获取消息成功；如果是 NULL，则使任务进入等待状态，并引发一次任务调度。

  • OSMboxAccept()

  • OSMboxQuery()

消息队列

消息队列由三个部分组成：事件控制块、队列控制块和消息指针。

消息队列的事件控制块：

成员 OSEventType 的值置为 OS_EVENT_TYPE_Q。

成员 OSEventPtr 指向了一个叫做队列控制块(OS_Q)的结构，该结构管理了一个数组 MsgTbl[]，该数组中的元素都是一些指向消息的指针；

消息队列相当于一个共用一个任务等待列表的消息邮箱数组。

  • 可移动的指针为 OSQIn 和 OSQOut，其中一个 移动到数组末尾也就是与 OSQEnd 相等时，会被调整到数组的起始位置 OSQStart；

  • 指针 OSQStart 和 OSQEnd 只是一个标志(常指针)；

  • 所以这个消息指针数组是一个环形的数据缓冲区。

每一个消息队列都要用到一个队列控制块，队列控制块的总数由 OS_CFG.H 中的常量 OS_MAX_QS 定义；

在系统初始化时，所有的队列控制块被成员 OSQPtr 链接成一个单向链表，即空队列控制块链表 —— OSQFreeList；

  • OSQCreate()：创建消息队列，调用这个函数之前首先需要定义一个指针数组，然后把各个消息数据缓冲区的首地址存入这个数组中；
    start：指针数组的地址，size：指针数组的长度。
    OS_EVENT OSQCreate(void** start, INT16U size );
    执行步骤
    从空闲事件控制块链表(OSEventFreeList)中取得一个 ECB；
    从空闲队列控制块列表(OSQFreeList)中取出一个队列控制块，并对其进行初始化；
    初始化 ECB 的内容(事件类型、等待任务列表)，并将 OSEventPtr 指针指向队列控制块。

  • OSQPend()：请求消息队列，目的是为了从消息队列中获取消息；
    如果消息队列中有消息，返回消息地址；如果没有，相应任务进入等待状态；

  • OSQAccept()：无等待地请求消息队列中的消息;
    如果消息队列中有消息，返回消息的地址；如果消息队列中没有消息，返回 NULL。
    void *OSQAccept(OS_EVENT *pevent);

  • OSQPost()：向消息队列发送消息，以 FIFO （先进先出）的方式组织消息队列；

如果有任务在等待该消息队列，唤醒其中优先级最高的任务，并重新调度；
如果没有任务在等待该消息队列，而且此时消息队列未满，则以 FIFO 方式插入这个消息。

  • OSQPostFront()：向消息队列发送消息，以 LIFO （后进先出）的方式组织消息队列；

信号量集

在实际应用中，任务常常需要与多个事件同步，即要根据多个信号量组合作用的结果来决定任务的运行方式。

μC/OS-II 为了实现多个信号量组合的功能定义了一种特殊的数据结构 —— 信号量集。

信号量集所能管理的信号量都是一些二值信号，所有信号量集实质上是一种可以对多个输入的逻辑信号进行基本逻辑运算的组合逻辑。

μC/OS-II 不使用事件控制块来描述信号量集，而使用了一个叫做标志组的结构 OS_FLAG_GRP。

  • OSFlagFreeList：空信号量集标志组链表，通过 OSFlagWaitList 指针链起来，共 OS_MAX_FLAGS 个空标志组。

  • OSFlagCreate()：创建信号量集；

  • OSFlagPend()：请求信号量集；

  • OSFlagPost()：向信号量集发信号；
    任务向信号量集发信号，就是对信号量集标志组中的信号进行置 1 或置 0 的操作。至于对信号量集中的哪些信号进行操作，用函数中的参数 flags 来指定；对指定的信号是置 1 还是置 0，用函数中的参数 opt 来指定 (opt = OS_FLAG_SET 为置 1 操作；opt = OS_FLAG_CLR 为置 0 操作)

等待任务链表

信号量集用一个双向链表来组织等待任务，每一个等待任务都是该链表中的一个节点(Node)。标志组 OS_FLAG_GRP 的成员 OSFlagWaitList 就指向了信号量集的这个等待任务链表。

信号量集的逻辑运算及控制：

发送信号量集的任务 -> OSFlagFlags -> OSFlagNodeWaitType -> 1/0? -> AND/OR? -> 请求信号量集的任务；

  • OS_FlagBlock()：给等待任务链表添加节点；
    在请求信号量集函数 OSFlagPend() 中被调用。

  • OS_FlagUnlink()：从等待任务链表删除结点；
    在发送信号量集函数 OSFlagPost() 中被调用。
    void OS_FlagUnlink (OS_FLAG_NODE *pnode);

内存的动态分配

应用程序在运行中为了某种特殊需要，经常需要临时获得一些内存空间，因此作为一个比较完善的操作系统必须具有动态分配内存的能力。

能否合理、有效地对内存储器进行分配和管理，是衡量一个操作系统品质的指标之一。特别地对于实时操作系统来说，还应该保证系统在动态分配内存时，它的执行时间必须是可确定的。

μC/OS-II 改进了 ANSIC 用来动态分配和释放内存的 malloc() 和 free() 函数，使它们可以对大小固定的内存块进行操作，从而使 malloc() 和 free() 函数的执行时间成为可确定的，满足了实时操作系统的要求。

内存控制块

μC/OS-II 对内存进行两级管理，即把一个大片连续的内存空间分成了若干个分区，每个分区又分成了若干个大小相等的内存块来进行管理。

操作系统以分区为单位来管理动态内存，而任务以内存块为单位来获得和释放动态内存。内存分区及内存块的使用情况则由表 —— 内存控制块来记录。

可动态分配内存的划分：

应用程序如果要使用动态内存的话，则要首先在内存中划分出可以进行动态分配的区域，这个划分出来区域叫做内存分区，每个分区要包含若干个内存块。

μC/OS-II 要求同一个分区中的内存块的字节数必须相等，而且每个分区与该分区的内存块的数据类型必须相同。

在内存中划分一个内存分区与内存块的方法非常简单，只要定义一个二维数组就可以了，其中的每个一维数组就是一个内存块。

例如，定义一个用来存储 INT16U 类型数据，有 10 个内存块，每个内存块长度为 10 的内存分区的代码如下：
INT16U IntMemBuf[10][10];

上面这个定义只是在内存中划分出了分区及内存块的区域，还不是一个真正的可以动态分配的内存区。只有当把内存控制块与分区关联起来之后，系统才能对其进行相应的管理和控制，它才能是一个真正的动态内存区。

  • OS_MEM：内存控制块，使系统能够感知和有效地管理内存分区，系统用这个来记录和跟踪每一个内存分区的状态。

  • OSMenFreeList：空内存控制块链表，用成员 OSMemFreeList 链起来，共 OS_MAX_MEM_PART 个内存控制块。

动态内存的管理

  • OSMemCreate()：建立内存分区；

  • OSMemGet()：向内存分区请求一个内存块；

  • OSMemPut()：释放一个内存块；

  • OSMemQuery()：查询一个分区目前的状态信息；

  • OS_MEM_DATA：上面 pdata 的结构类型
